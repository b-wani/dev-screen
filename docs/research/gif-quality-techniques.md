# GIF 화질 개선 기법 조사 (#120)

recap의 GIF export 경로(`src/renderer/src/export.ts` → `renderRecipeToGif` → `gifenc`)에서
256색 제약 안의 화질을 끌어올릴 수 있는 수단을 조사하고, 각 기법의 적용 가능성과
화질/용량 트레이드오프를 정리한다. 프로파일 매트릭스(#121)의 색·디더링 축 입력이 된다.

**1차 소스**: 로컬에 설치된 gifenc **1.0.3** 소스(`node_modules/gifenc/`) — `README.md`,
`src/palettize.js`(`applyPalette`/`prequantize`), `src/pnnquant2.js`(`quantize`),
`src/index.js`(export 목록). 버전이 lockfile에 고정돼 있어 라이브러리 자체가 최고 신뢰 소스다.

## 요약 (결론 먼저)

| 기법 | gifenc 지원 | 화질 효과 | 용량 영향 | 적용 난이도 | 권고 |
|------|:----------:|----------|----------|:----------:|------|
| **프레임별 로컬 팔레트** | ✅ (이미 사용 중) | 높음 | +소폭 | — (현행) | 유지 |
| **디더링** | ❌ **미지원** | (구현 시) 그라디언트만 개선 | **+대폭(노이즈로 압축 악화)** | 높음(직접 구현) | 스코프 밖 권고 |
| **차분/투명 프레임** | ✅ (`transparent`/`dispose`) | 중립 | **-대폭(정적 화면일수록)** | 높음(글로벌 팔레트 필요) | 별도 검토 |
| **해상도 상향** | ✅ (앱 측 캔버스) | **매우 높음(텍스트/UI)** | +대폭(픽셀 수 비례) | 낮음(설정값) | **1순위 축** |
| quantize `format`/`useSqrt` | ✅ | 소폭 | 중립 | 낮음 | 현행 기본값(rgb565)이 최적 |

**핵심 반전 2가지:**
1. **gifenc에는 디더링이 아예 없다.** README가 명시("there is currently no dithering support")하고,
   export 목록에도 디더링 함수가 없으며 `applyPalette`는 순수 최근접 색 매핑이다.
   → **#121의 "디더링 축"은 gifenc 옵션으로 소싱 불가.** 직접 구현하거나 축에서 뺀다(스코프 결정).
2. **화질의 가장 큰 레버는 색/디더링이 아니라 해상도와 fps다.** 현행 `GITHUB_PRESET`이 480p/15fps로
   묶여 있어, 720p/1080p 상향과 fps 상향(#119: 50fps 상한)이 체감 화질을 가장 크게 올린다.

## 현재 코드가 이미 하고 있는 것

`renderRecipeToGif`(`export.ts:114–127`)는 프레임마다:

```js
const palette = quantize(data, config.maxColors)   // 프레임별 로컬 팔레트 (rgb565, useSqrt 기본)
const index = applyPalette(data, palette)          // 최근접 매핑, 디더링 없음
encoder.writeFrame(index, w, h, { palette, delay }) // 로컬 색테이블을 프레임마다 기록
```

즉 **프레임별 로컬 팔레트는 이미 화질-최대 설정**으로 쓰고 있고, quantize도 기본값
rgb565(5-6-5비트, 고품질)·`useSqrt`(지각 가중, 기본 on)를 쓴다. 색 축에서 짜낼 여지는 크지 않다.

## 축별 상세

### 1. 프레임별 로컬 팔레트 — 이미 최적, 튜닝 여지 낮음

- **지원**: `writeFrame`의 `palette` 옵션이 프레임별 로컬 색테이블을 기록한다(README "per frame color palette").
- **현행**: 프레임마다 `quantize`로 그 프레임 콘텐츠에 맞춘 256색을 새로 뽑는다. 화질상 최선.
- **대안(글로벌 팔레트)**: `quantize`를 한 번만 하고 전 프레임이 공유. 프레임당 색테이블(~최대 768B)을
  안 실어 용량이 줄지만, 장면 전환이 있는 화면 녹화에선 화질이 떨어진다.
- **트레이드오프**: 로컬=화질↑·용량 소폭↑, 글로벌=용량↓·화질↓ + **차분 프레임의 전제조건**(아래 3번).
- **결론**: 화질 목적지에선 현행(로컬) 유지. 이 축은 "개선 여지"가 아니라 이미 켜져 있는 스위치다.

### 2. 디더링 — gifenc **미지원**, 직접 구현은 화면 녹화에 역효과 가능성

- **지원**: **없음.** `applyPalette`(`palettize.js:38`)는 각 픽셀을 팔레트의 유클리드 최근접 색으로
  바꿀 뿐 오차 확산/순서 디더가 없다. gifenc README도 "no dithering support"를 한계로 명시하고,
  "future version"의 미구현 항목으로 나열한다.
- **직접 구현 시**: Floyd–Steinberg(오차 확산) 또는 Bayer(순서 디더)를 `applyPalette` 대신 앱에서 구현하고
  `nearestColorIndex`(gifenc가 export함)로 팔레트 인덱스를 찾는 방식은 가능하다. 단:
  - **용량 대폭 증가**: 디더는 고주파 노이즈를 넣어 LZW 압축률을 떨어뜨린다 → 파일이 커진다.
  - **화면 녹화 특성**: 평면 UI·텍스트에는 노이즈만 추가돼 오히려 지저분해진다. 이득은 사진/그라디언트/
    부드러운 셰이딩 영역에 한정된다. recap의 주 콘텐츠(앱 화면)와 상성이 나쁘다.
  - CPU 비용 증가(프레임당 오차 확산 루프).
- **결론**: **디더링 축은 스코프에서 빼는 것을 권고.** #121에서 굳이 넣는다면 "gifenc 외 직접 구현"이라는
  별도 구현 티켓과 화면 녹화 대상 A/B가 선행돼야 하며, 화질 이득 대비 용량·복잡도 손해가 크다.

### 3. 차분/투명 프레임(delta frames) — 용량 최대 절감 레버, 단 구조 변경 필요

- **지원**: `writeFrame`의 `transparent`(1비트 투명)·`transparentIndex`·`dispose` 옵션으로 가능
  (README API). 이전 프레임과 안 변한 픽셀을 투명 인덱스로 칠하고 `dispose`를 "화면 유지"로 두면,
  변한 영역만 다시 그린다.
- **효과**: 화면 녹화는 프레임 간 정적 영역이 크다 → 투명 픽셀의 긴 런이 LZW로 매우 잘 압축돼
  **용량이 크게 줄어든다**(정적일수록 이득↑). 화질 자체는 중립(색은 그대로).
- **제약/난이도**:
  - **글로벌(또는 최소한 고정 투명 인덱스) 팔레트가 사실상 필요** — 프레임별 팔레트는 프레임마다
    인덱스 의미가 바뀌어 투명 인덱스를 고정하기 어렵다. → 2번(로컬 팔레트)과 상충, 구조 재설계.
  - **양자화 노이즈로 "안 변한" 픽셀도 미세하게 달라짐** → 픽셀 동일성 임계값(threshold) 필요.
    임계를 낮추면 절감 미미, 높이면 잔상(고스팅)이 남는다.
  - `dispose` 모드(1=유지) 설정을 프레임마다 정확히 다뤄야 한다.
- **결론**: **화질이 아니라 용량 축**의 강력한 수단. 맵이 용량을 "하드캡→경고 임계"로 재정의(#118)했으므로
  최우선은 아니지만, 고해상도·고fps로 용량이 커질 때 이를 상쇄하는 카드로 **별도 티켓 후보**(Not yet
  specified). 로컬 팔레트 화질과 트레이드오프이므로 프로파일 확정 시 함께 판단.

### 4. 해상도 상향 — 화질의 1순위 레버, 앱 측 설정

- **지원**: gifenc와 무관. `resolveGifConfig`(`export-preset.ts:117–120`)의 캔버스 크기 문제다.
- **효과**: 480p→720p→1080p는 텍스트·UI 선명도를 가장 크게 올린다(256색 제약과 독립).
- **용량**: 대략 픽셀 수에 비례(720p ≈ ×2.25, 1080p ≈ ×5 원시 기준). 단 평면 UI는 LZW 압축이
  잘 먹어 실측 증가는 sublinear인 경우가 많다 — 정확한 곡선은 #121 프로파일 매트릭스 실측으로.
- **중요 제약**: 현행 `resolveGifConfig`는 `Math.min(1, gifMaxHeight/source.height)`로 **원본보다
  키우지 않는다.** 즉 480p로 **녹화된** 원본은 1080p GIF로 못 만든다 — 상향 실효는 원본 해상도가
  충분할 때만 발생. (녹화 파이프라인 해상도는 이 티켓 범위 밖.)
- **결론**: 색/디더링보다 우선하는 화질 축. #121 매트릭스의 사실상 주축으로 삼을 것.

## #121(프로파일 매트릭스)로 넘기는 입력

- **색 축(실제 gifenc 노브)**: `maxColors`(≤256), quantize `format`(rgb565 기본=고품질 / rgb444=저품질·고속),
  `useSqrt`(기본 on). 현행이 이미 고품질 조합 → 색 축은 "256색 유지"가 사실상 기본, 튜닝 이득 작음.
- **디더링 축**: **gifenc 옵션으로 존재하지 않음.** 축을 유지하려면 "직접 구현" 스코프 결정이 선행돼야 하며,
  화면 녹화 대상엔 용량·화질 모두 불리 → **축에서 제외 권고.**
- **주 화질 축은 해상도 + fps(#119: 50fps 상한)** 로 재편하는 것이 실익이 크다.
- **용량 상쇄 카드**: 차분/투명 프레임(글로벌 팔레트 전환 필요) — 화질 축이 아니라 용량 축의 별도 티켓 후보.
