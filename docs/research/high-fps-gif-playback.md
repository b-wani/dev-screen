# 고fps GIF 실효 재생 검증 (#119)

recap의 GIF export 경로(`src/renderer/src/export.ts` → `gifenc`)가 요청 fps를 실제로 어떻게
파일에 기록하는지, 그리고 뷰어가 그 타이밍을 존중하는지 검증한다. 프로토타입 산출물이며,
동일 gifenc 호출 경로를 그대로 재현한 probe 스크립트(`gif-fps-samples/probe.mjs`)와
샘플 GIF 4종(`gif-fps-samples/sample-{15,30,50,60}fps.gif`)이 근거다.

## 결론

- **recap의 최대 실효 fps = 50fps.** 가설 확정.
- **60fps는 근사조차 불가** — 요청해도 파일에는 50fps로 기록된다.

## 왜 50fps가 상한인가 (인코더 산술)

GIF 프레임 딜레이는 **1/100초(센티초) 정수** 단위로만 저장된다. gifenc는
`writeFrame({ delay })`의 ms 값을 다음처럼 반올림한다
(`node_modules/gifenc/dist/gifenc.esm.js`, `writeFrame` 내부):

```js
let z = Math.round(delayMs / 10)   // ms → 센티초, 반올림
```

recap은 `frameDelayMs = 1000 / fps`를 넘기므로(`export.ts:106,125`), 실효 fps는
`1000 / (round((1000/fps)/10) * 10)`으로 결정된다. probe로 생성한 GIF의 GCE(그래픽 제어 확장)
delay 바이트를 역파싱한 실측값:

| 요청 fps | 요청 delay | 기록 delay | 실효 fps | 비고 |
|---------|-----------|-----------|---------|------|
| 15 (현행 `GITHUB_PRESET`) | 66.67ms | **7cs (70ms)** | **14.3** | 반올림으로 의도보다 느림 |
| 30 | 33.33ms | **3cs (30ms)** | **33.3** | 의도보다 살짝 빠름 |
| 50 | 20.00ms | **2cs (20ms)** | **50.0** | 정확히 일치 |
| 60 | 16.67ms | **2cs (20ms)** | **50.0** | 60→50 붕괴 |

타이밍이 정확히 맞는 fps는 센티초의 약수 계열뿐이다: **100 / 50 / 33.3 / 25 / 20 / …fps**.
그 외(60, 45, 40 등)는 가장 가까운 센티초로 반올림돼 어긋난다.

### 60fps의 추가 함정: 재생 시간 늘어짐

`export.ts`는 `totalFrames = round(durationSec × fps)`로 프레임 수를 뽑는다. 60fps로 뽑으면
50fps 대비 프레임이 1.2배 많은데 각 프레임이 20ms로 재생되므로, **원본 대비 재생 길이가
1.2배로 늘어난다(20% 슬로모션)**. 즉 60fps 요청은 프레임 상한만 막는 게 아니라 타이밍을
왜곡한다 — UI/프리셋에서 60fps는 노출하지 않는 편이 안전하다.

## 뷰어 재생 (저딜레이 클램프)

브라우저는 역사적으로 **delay ≤ 1cs(≈10ms)** 인 프레임을 100ms로 강제 클램프한다.
gifenc는 위 범위(≤50fps)에서 **최소 2cs(20ms)** 미만을 절대 쓰지 않으므로 클램프 위험 구간에
진입하지 않는다 — 50fps(2cs)는 그대로 재생된다.

- **Chromium**: ✅ 로컬 HTTP로 4종 로드, 모두 정상 애니메이션. 2/3/7cs delay 존중.
- **macOS 미리보기(Preview)**: ✅ 애니메이션 재생, 고fps가 눈에 띄게 더 부드러움 — delay 존중 확인.
- **Dooray 본문 인라인 삽입**: ✅ 자동재생되며 고fps가 더 부드럽게 재생 — 저딜레이 클램프 없음 확인.

세 뷰어 모두 delay를 존중하며 고fps일수록 부드럽다. Dooray 인라인 자동재생·고fps 재생이 성립함이
확정됐고(#118에서 넘어온 미해결 항목 해소), GIF로 고품질 첨부가 가능하다는 목적지 전제도 유지된다.
