# Dooray GIF 첨부 제약 실측

- 이슈: #118 (맵: #117)
- 조사일: 2026-07-15
- 조사 대상: Dooray 공식 helpdesk 가이드(`helpdesk.dooray.com`)의 파일 업로드·업무 등록/수정·용량 초과 제한·이미지 사이즈 조절 페이지, Dooray 서비스 API 레퍼런스 문서, 실제 테넌트 계정(`bwkim@nhn.com`, org `1387695619080878080`) MCP 조회
- 방법: WebSearch로 공식 helpdesk 페이지 후보 수집 → Playwright로 SPA 렌더링 후 본문 원문 추출(helpdesk는 JS 렌더 SPA라 WebFetch로는 본문이 안 잡힘) → API 레퍼런스는 원문 텍스트 추출 후 키워드 검색 → MCP `member_me`로 계정 확인. 실제 업로드 테스트는 하지 않음(사유는 하단 (a) 참고).

## 요약

- Dooray는 **"업무 본문/댓글에 이미지 삽입"** 과 **"업무에 파일 첨부"** 를 공식적으로 별개 기능으로 구분한다. 즉 GIF를 본문 인라인 이미지로 넣는 것과 파일로 첨부하는 것은 동작·표시가 다르다. [용량 초과 제한]
- 공식 문서에서 확인되는 **유일한 per-file 용량 수치는 드라이브 업로드 기준**(Basic 이하 500MB, Business 이상 10GB)이며, **업무/댓글 첨부 전용 상한 수치는 공식 문서에서 명시 확인 불가**다. 업무 첨부는 사실상 조직의 잔여 용량(quota)에 종속되어, 용량 초과 시 "업무에 파일 첨부 불가"로 막힌다. [파일 업로드][용량 초과 제한]
- 본문 인라인 이미지는 스마트 에디터/마크다운(`<img>`)으로 삽입되며 뷰어에서 인라인 표시된다. GIF를 인라인으로 넣으면 브라우저 `<img>` 표준 동작상 애니메이션이 재생되는 것이 자연스럽지만, **Dooray가 GIF 자동재생/재생 성능을 공식 문서에서 언급한 근거는 찾지 못했다(추정)**. [이미지 사이즈 조절]
- 고fps·대용량 GIF 재생 정상성, WebP/APNG 애니메이션 지원 여부는 **공식 문서에 언급이 전무**하여 확인 불가.
- recap 관점: 하드 용량 상한(≥500MB)은 GIF에 비해 매우 크므로 용량 경고 임계는 하드캡이 아니라 **인라인 렌더 부담/뷰어 재생 UX 기준**으로 잡는 것이 타당하다.

## (a) 첨부 가능한 최대 파일 크기 상한

**결론: 업무 첨부 전용 상한 수치는 공식 문서에서 명시 확인 불가. 확인된 유일한 공식 per-file 수치는 드라이브 기준.**

- 드라이브 파일 업로드 가이드 원문: "1개당 최대 업로드 파일 크기는 Basic 이하는 500MB, Business 이상은 10GB 까지 가능합니다." 단 같은 페이지에서 "업무나 위키의 첨부 파일이 드라이브에 보이지는 않습니다"라고 명시하여, 이 수치가 곧 업무 첨부 상한이라고 단정할 근거는 없다. [파일 업로드]
- 업무 등록/수정 가이드에는 첨부파일의 용량 상한·개수 제한·허용 확장자에 대한 문구가 전혀 없다. [업무 등록/수정]
- Dooray 서비스 API 레퍼런스의 첨부 업로드 스펙(`POST /project/v1/post-drafts/{post-draft-id}/files`, `Content-Type: multipart/form-data`)에도 최대 크기·허용 포맷 수치가 명시돼 있지 않다(문서 전문에서 MB/GB/용량 관련 수치 검색 결과 없음). [Dooray API]
- 용량 초과 제한 표 원문: 개인/공용 용량 초과 시 "업무에 파일 첨부 불가", "업무 본문/댓글에 이미지 삽입 불가"로 막힌다. 즉 업무 첨부의 실질 상한은 **조직 잔여 용량(quota)** 이 결정하며, 별도의 고정 per-file 캡은 문서화돼 있지 않다. [용량 초과 제한]
- 실측 미수행 사유: 이 세션의 MCP `task_file_upload`는 **에이전트가 아닌 MCP 서버 호스트의 파일시스템**에서 파일을 읽는 구조라(서버 안내 문구) 임의 크기의 테스트 GIF를 서버 호스트에 배치할 수 없어 업로드 상한을 에러 메시지로 실측하는 것이 불가능했다. 또한 실 테넌트에 테스트 첨부를 남기지 않기로 하여 시도하지 않았다. [MCP 서버 안내]

## (b) 인라인 렌더/자동 재생 여부

**결론: 본문/댓글 "이미지 삽입"은 인라인 표시가 공식 지원됨(확인). 파일로 "첨부"만 하면 인라인 자동재생이 아니라 첨부 목록 항목이다(구분 확인). GIF의 애니메이션 자동재생 자체는 공식 근거 없음(추정).**

- Dooray는 업무에서 두 경로를 명시적으로 구분한다. 용량 초과 제한 표가 "업무 본문/댓글에 이미지 삽입"과 "업무에 파일 첨부"를 별개 행으로 다룬다 — 이는 인라인 이미지와 파일 첨부가 서로 다른 기능임을 방증한다. [용량 초과 제한]
- 본문 인라인 이미지 삽입 경로가 공식 문서화돼 있다: 스마트 에디터에서 이미지를 클릭하면 "크기 조정" 툴바가 뜨고, 마크다운 모드에서는 `<img>` 태그에 `src`/`width`를 직접 넣어 삽입한다. 즉 본문에 넣은 이미지는 뷰어에서 인라인으로 렌더된다. [이미지 사이즈 조절]
- GIF를 이 인라인 경로(`<img>`)로 삽입하면 브라우저 표준상 애니메이션이 재생되는 것이 자연스럽다. 그러나 **Dooray 문서가 "GIF가 본문에서 움직여 보인다/자동재생된다"고 명시한 문구는 찾지 못했다** — 따라서 "인라인 삽입 시 재생됨"은 확인이 아니라 추정으로 남긴다. [근거 부재]
- 반대로 파일을 첨부(attachFile)만 한 경우, 문서상 이는 첨부 목록의 다운로드 대상이며 본문에서 자동으로 움직이는 미리보기가 보장된다는 근거는 없다. [근거 부재]

## (c) 고fps·대용량 GIF 재생 정상 여부

**결론: 확인 불가. 공식 문서에 재생 성능/프레임레이트 관련 언급이 전혀 없음.**

- helpdesk·API 어디에도 GIF 프레임레이트, 재생 속도, 대용량 애니메이션 처리에 대한 기술 문구가 없다. [근거 부재]
- 인라인 이미지가 브라우저 `<img>`로 렌더되는 구조라면 재생 속도·부하는 사실상 뷰어(클라이언트 브라우저/앱)의 성능에 종속되며 Dooray가 재생 속도를 보정한다는 근거는 없다(추정). 의도한 fps 그대로 재생되는지는 실제 뷰어에서 눈으로 확인해야 하며, 이번 조사에서는 실 테넌트 렌더 확인을 하지 못했다. [근거 부재]

## (d) 부수: 애니메이션 WebP/APNG 지원 여부

**결론: 확인 불가/추정. 공식 문서에서 WebP·APNG 첨부/인라인 지원 명시를 찾지 못함.**

- helpdesk·API 문서에 WebP, APNG 포맷에 대한 언급이 없다. 지원 확장자 화이트리스트를 명시한 공식 페이지도 확인되지 않았다. [근거 부재]
- 최신 브라우저는 애니메이션 WebP/APNG를 `<img>`로 렌더하므로, 본문 인라인 삽입 시 표시될 가능성은 있으나 이는 브라우저 능력에 기댄 추정이며 Dooray 공식 근거는 없다. recap이 WebP/APNG를 산출물로 고려한다면 별도 실측이 필요하다. [근거 부재]
